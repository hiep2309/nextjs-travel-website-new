# VN Insight — Multilingual Architecture (next-intl)

Production i18n for **vi** (default), **en**, **ko**.

**Rule:** UI strings → `messages/*.json` via next-intl. Dynamic content → Firestore `{ vi, en, ko }` maps via `getTranslation()`.

---

## Phase 1 — next-intl foundation (done)

```
i18n/
  routing.ts       # locales, LOCALE_COOKIE, localePrefix: always
  request.ts       # loads messages/{locale}.json
  navigation.ts    # re-exports locale-aware Link, useRouter

middleware.ts      # locale detection + cookie persistence + hreflang links

app/[locale]/
  layout.tsx       # html lang, NextIntlClientProvider, root hreflang
  */layout.tsx     # client routes: initPageLocale + generateMetadata
  */page.tsx       # initPageLocale on every server page

lib/
  getTranslation.ts    # getTranslation(field, locale) — DB content only
  i18n/
    server.ts          # initPageLocale(), parseLocale()
    pageMetadata.ts    # generatePageMetadata(), PAGE_META
    content.ts         # pickLocalized (internal)
    metadata.ts        # buildLocalizedMetadata + hreflang
    types.ts           # LocalizedString, MultilingualPost, …

components/LocaleSwitcher.tsx  # animated glass dropdown, route-preserving
```

Every `[locale]` page calls `initPageLocale(params.locale)` for static rendering. Client-only routes use a thin `layout.tsx` wrapper.

## Phase 2 — Multilingual database schema (done)

Canonical Firestore shape for `posts/{id}`:

```json
{
  "sourceLocale": "vi",
  "title": { "vi": "…", "en": "…", "ko": "…" },
  "description": { "vi": "…", "en": "…", "ko": "…" },
  "contentHtml": { "vi": "<p>…</p>", "en": "…", "ko": "…" },
  "slugs": { "vi": "ha-noi-abc", "en": "hanoi-abc", "ko": "…" },
  "seo": { "vi": { "title": "…", "description": "…" }, "en": { … } },
  "translationStatus": { "vi": "published", "en": "machine", "ko": "machine" }
}
```

**Helpers** (`lib/firestore/multilingual.ts`):

| Function | Purpose |
|----------|---------|
| `normalizeTravelPost(id, raw)` | Legacy string → `{ vi }` maps on read |
| `buildPostLocaleWritePayload()` | Create/update write bundle |
| `flattenLocalizedForSearch()` | Search across all locales |
| `getTranslation(field, locale)` | Render active locale (`lib/getTranslation.ts`) |

**Migration** (existing documents):

```bash
node scripts/migrate-posts-localized.mjs --dry-run
node scripts/migrate-posts-localized.mjs
```

Legacy `name` / `slug` string fields are normalized on read; new saves omit them.

## 2. URL routing

| Locale | Example |
|--------|---------|
| vi | `/vi/destinations/ha-noi` |
| en | `/en/destinations/ha-noi` |
| ko | `/ko/destinations/ha-noi` |

- **Fallback:** Vietnamese (`vi`)
- **Detection:** `Accept-Language` via middleware
- **Prefix:** always (`localePrefix: 'always'`)

Use navigation helpers — never hardcode `/vi/...`:

```tsx
import { Link, useRouter } from "@/i18n/navigation";

<Link href="/explore">{t("destinations")}</Link>
router.push("/profile");
```

## 4. Firebase multilingual schema

Store translations as **maps keyed by locale** (not separate documents per language).

### `posts/{id}`

```json
{
  "sourceLocale": "vi",
  "title": { "vi": "Hà Nội 3 ngày", "en": "3 days in Hanoi", "ko": "하노이 3일" },
  "description": { "vi": "...", "en": "...", "ko": "..." },
  "contentHtml": { "vi": "<p>...</p>", "en": "<p>...</p>" },
  "slugs": { "vi": "ha-noi-3-ngay", "en": "hanoi-3-days", "ko": "hanoi-3-il" },
  "seo": {
    "vi": { "title": "...", "description": "...", "keywords": ["hà nội"] },
    "en": { "title": "...", "description": "..." }
  },
  "translationStatus": {
    "vi": "published",
    "en": "reviewed",
    "ko": "machine"
  }
}
```

### Resolve in code

```tsx
import { getTranslation } from "@/lib/getTranslation";
import { useLocale } from "next-intl";

const locale = useLocale();
const title = getTranslation(post.title, locale);
```

Legacy alias: `pickLocalized()` / `pickLocalizedHtml()` from `@/lib/i18n/content`.

Legacy string fields still work — `getTranslation` falls back to Vietnamese automatically.

}

## 7. AI-assisted translation workflow

**Helper module:** `lib/translation/` (public API via `@/lib/translation` or `@/lib/i18n`).

| Function | Where | Purpose |
|----------|-------|---------|
| `translateText()` | Server | Gemini → MyMemory fallback |
| `translateTextClient()` | Browser | Calls `/api/translate` |
| `buildLocalizedString()` | Browser | Create Post — `{ vi, en, ko }` |
| `buildLocalizedStringServer()` | Server/scripts | Same, no HTTP |
| `translatePostFields()` | Server/admin | Title + description + HTML bundle |
| `runPostTranslationPipeline()` | Server | Full Firestore write payload |
| `requestPostTranslation()` | Browser | POST `/api/translate/post` |

```ts
import { requestPostTranslation } from "@/lib/translation";

const payload = await requestPostTranslation({
  title: "Hà Nội 3 ngày",
  description: "Lịch trình gợi ý…",
  contentHtml: "<p>…</p>",
  sourceLocale: "vi",
});
// payload → { title, description, contentHtml, slugs, seo, translationStatus, sourceLocale }
```

**API routes**
- `GET/POST /api/translate` — single/batch text chunks
- `POST /api/translate/post` — full post pipeline (Create Post + Admin dashboard)

**Batch backfill** (dev server must be running):

```bash
node scripts/translate-posts-ai.mjs --dry-run
node scripts/translate-posts-ai.mjs
node scripts/translate-posts-ai.mjs --id=POST_ID
```

1. Author writes in **source locale** (usually `vi`).
2. Export JSON fields → send to LLM with glossary (place names, brand “VN Insight”).
3. Save as `translationStatus: "machine"`.
4. Human editor reviews in admin → `"reviewed"` → `"published"`.
5. Never auto-translate on the live site at request time (latency + SEO).

**Prompt template:**

```
Translate the following travel content from Vietnamese to {targetLocale}.
Keep proper nouns (Hạ Long, Sa Pa). Output JSON only.
Fields: title, description, contentHtml (preserve safe HTML tags).
```
